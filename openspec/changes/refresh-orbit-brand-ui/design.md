# Design

## Visual Thesis

- Orbit 控制台：一块面向本地 LLM 工作区的桌面控制台，气质克制、清晰、偏工具而非营销；视觉重心来自排版、留白和状态层级，而不是装饰色块。

## Content Plan

- 主窗口：账号轨道 -> CLI 主工作区 -> 账号检视区 -> 危险区。
- 新增账号窗口：模式选择 -> 当前步骤说明 -> 当前模式字段 -> 固定操作条。

## Interaction Thesis

- `selection transition`：左侧选中账号与中央工作区内容做快速平滑切换，强调“切换上下文”。
- `hover affordance`：账号行、最近目录和关键次级操作只用轻微底色或轮廓变化提示可点，不做浮夸缩放。
- `sheet mode transition`：新增账号窗口在不同模式间只切换当前表单块，保持标题和操作区稳定。

## Current Diagnosis

- 主窗口首屏没有唯一主动作，`打开 CLI` 只是多个同级块之一。
- 详情、额度、状态都使用相近的卡片形态，导致阅读顺序依赖用户自行判断。
- 左侧列表和右侧详情都偏厚重，整个窗口像多个面板拼接，而不是一个连续工作台。
- 新增账号窗口把解释、状态、字段和主按钮堆在一列里，切换模式时缺少“当前正在完成哪一步”的感知。

## Design Principles

- 先定构图，再定组件；先定主次，再定装饰。
- 首屏只能有一个主任务，其他内容都为主任务服务或提供校验。
- 默认去卡片化；只有交互容器或状态提醒才使用明显 surface。
- 文案优先操作语言，不写营销式说明，不重复解释同一件事。
- Orbit 品牌必须在首屏可辨认，但不能压过当前任务。
- 不使用网页式 hero、装饰性渐变、多强调色或厚重边框。

## Design System

### Color Roles

- `background`：窗口整体背景，接近系统背景色，用于维持原生感。
- `sidebar`：左侧账号轨道背景，比 `background` 略深，用来收束导航区域。
- `workspace`：中央主工作区背景，视觉上最稳定、最干净。
- `surface`：次级区块、提醒、嵌入式表单容器。
- `accent`：唯一强调色，用于主按钮、选中态和关键链接。
- `success` / `warning` / `danger`：仅用于状态反馈和危险操作。

### Typography Roles

- `display`：产品名或窗口级标题，控制在一屏一个焦点。
- `sectionTitle`：主区块标题，如“打开 CLI”“账号详情”。
- `body`：常规字段、表单、按钮文案。
- `meta`：路径、说明、状态补充、次要标签。

### Spacing Rhythm

- 统一使用 `8 / 12 / 16 / 24 / 32`。
- 大块之间至少 `24`，同一语义组内部优先 `8` 或 `12`。

### Radius

- `10`：紧凑交互行，如最近目录、内嵌状态块。
- `14`：常规 surface。
- `20`：仅保留给主工作区或模式容器，不泛滥使用。

### Surface Rules

- 首屏最多一个高强调主 surface 和一个低强调次级 surface。
- 若移除背景、阴影、圆角后信息仍清晰，则默认不用卡片。
- 分隔优先用留白、对齐和弱分割线，不优先用大面积填色。

### Motion Rules

- 只保留 `selection transition`、`hover affordance`、`sheet mode transition`。
- 动效必须快速、克制、服务状态切换；不为“显得高级”而加入额外动画。

## Main Window

### Structure

- 采用 `账号轨道 / 主工作区 / 检视区` 三层结构。
- 左侧只负责选择账号、显示平台和当前状态，并承载少量全局动作，不再与详情区争主次。
- 中央作为唯一第一焦点，围绕“打开 CLI”组织目标选择、主按钮、最近目录和必要说明。
- 右侧作为 inspector，按“账号详情 -> 额度/限额 -> 状态 -> 路径 -> 删除账号”自上而下排列。

### Account Rail

- 账号行更紧凑、更高密度，强调名字、当前态、平台和一行状态摘要。
- 侧栏底部保留新增账号、刷新全部、语言与 home 路径，但视觉权重必须低于中央主工作区。
- 选中态依赖 accent 和背景层级，不依赖厚边框或大量胶囊标签。

### Workspace

- 顶部允许出现轻量品牌和当前账号上下文，但 `打开 CLI` 必须是最醒目的 `sectionTitle`。
- 主按钮保持整块宽度优势；目标选择器和帮助文案退为次级信息。
- 最近目录不再堆成“卡片画廊”，改为更接近列表或轻量可点击行。
- 次级动作如手动更新、切换账号、编辑供应商、重启 Codex 保持同组但降级，不与主按钮竞争。

### Inspector

- 账号详情、额度、状态和路径改成连续检视区，减少同级大卡片的数量。
- 每段只有一个职责和一个标题，避免同一屏出现多个外观相同但权重不明的色块。
- 删除账号沉到底部危险区，与常规检视信息明确分离。

### Empty and Warning States

- 空状态仍出现在右侧主区域，但文案只说明当前该做什么。
- 警告和错误使用 `warning` / `danger` surface，不和常规详情共用同样层级。

### Window Constraints

- 在 `960x620` 下，账号轨道、主按钮、最近目录首项和 inspector 起始信息必须同时可见。
- 不允许首屏因为横向拥挤而把主按钮、目标选择器或关键状态挤进折叠层。

## Add Account Window

### Structure

- 顶部固定窗口标题与关闭操作，下方是接入方式切换。
- 中部只渲染当前模式所需字段与说明，不保留其他模式的残余占位。
- 底部固定取消 / 主操作条，保证在滚动时始终明确下一步。

### Mode Behavior

- `ChatGPT 浏览器登录` 强调授权入口和回调提交，辅助文字只保留回调说明。
- `API Key Provider` 以规则、供应商和最小必填字段为主，自定义字段只有在当前 preset 允许时保持可编辑。
- `Claude Profile` 保持最轻量，突出“导入本地快照”，不制造伪登录感。

### Copy Rules

- 状态文案必须贴近当前步骤，例如“重新打开授权页面”“提交回调”“保存并激活供应商”。
- 解释文案放在字段附近，不在窗口顶部堆积长段说明。

### Visual Rules

- 模式容器允许一个主 surface；不同模式通过标题、说明和字段变化建立差异，不靠蓝色、橙色或强调色大面积铺底。
- 编辑 Provider 时继续沿用同一结构，只在必要位置显示“规则已锁定”“留空表示继续使用当前 API Key”。

## Menu Bar Consistency

- 菜单栏面板不重做结构，但后续实现要对齐新的颜色层级、选中态和行内密度，避免主窗口和菜单栏像两个产品。
